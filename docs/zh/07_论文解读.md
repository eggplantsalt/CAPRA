# 07 论文解读：CAPRA 做了什么，为什么这样做

> 本文面向想要理解 CAPRA 背后研究动机与方法论的读者。
> 不需要先读论文，本文就是"代码背后的论文解读"。

---

## 1. 问题是什么

### 1.1 现有 VLA 的安全盲区

当前的视觉-语言-动作模型（VLA）被训练为尽可能快速准确地完成任务，
完全不考虑副作用。

**典型场景：**
机器人被要求"把红色杯子放到架子上"，有两条路径：
- 路径 A：绕过旁边的花瓶，稍慢一点
- 路径 B：穿过花瓶旁边，更高效但可能碰倒花瓶

现有 VLA 倾向于路径 B——轨迹更短、更接近训练数据。
**花瓶是否倒下，不在损失函数里。**

### 1.2 为什么不用测试时安全滤波器

CAPRA 选择不在推理时加"安全检查器"，原因：
1. 安全判断本身就需要预测后果，是个复杂问题
2. 滤波器增加推理延迟，实时控制对延迟敏感
3. 根本解法是改变偏好，而不是每步拦截

**CAPRA 的目标：改变模型学到的偏好，测试时不加任何约束。**

## 2. 核心洞察

### 2.1 副作用只在"有选择"时才可避免

**关键洞察：只有在任务等价集 E_t 非空时，副作用才是可避免的。**

```
E_t = { 能以接近最优进度完成任务的候选动作集合 }
```

当 E_t 中有多个选项时，CAPRA 才介入，引导模型选足迹最小的那个。
E_t 为空时，CAPRA 损失不触发——不干扰模型的任务执行。

### 2.2 足迹是可分解的三分量

```
F_t(a) = alpha_d * D_t  +  alpha_i * I_t  +  alpha_r * R_t

  D_t : 非目标物体位移（精确）
  I_t : 接触冲量（近似）
  R_t : 不可逆事件：倾倒/支撑断裂（近似，权重最高）
```

分解的好处：可针对不同风险类型设置权重，
可以追踪每个物体的贡献，调试友好。

### 2.3 用 KL 散度传递安全偏好

```
L = L_anchor  +  lambda * sum_t  w_t * KL(q_hat_t || pi_theta_t)

  q_hat_t    : 安全目标分布，E_t 中足迹越小的动作概率越高
  pi_theta_t : 模型当前对 K 候选动作的评分分布
  w_t        : Delta_t * (1 + rho * R_t)  可避免风险越大权重越大
```

KL 散度把 pi_theta 拉向 q_hat，
L_anchor 确保任务性能不丢失。

## 3. 方法论的四个组件

### 组件一：离线挖掘

在线训练时不可能暂停仿真跑 K 次 rollout，所以离线化：

```
用 baseline 模型跑 episode，每步：
  1. 保存 MuJoCo 快照
  2. 采样 K 个候选动作（nominal + 高斯噪声扰动）
  3. 从快照出发各执行 H_s 步
  4. 记录 P_t（任务进度）和 F_t（足迹）
  5. 过滤等价集 E_t，计算安全目标分布 q_hat_t
  6. 存入缓存 JSON
训练时从缓存读取，无需重跑仿真。
```

### 组件二：任务等价集过滤

三重过滤，只在"有选择余地"时触发约束：

```
a_i 属于 E_t  iff：
  P_t(a_i) >= progress_floor          最低进度门槛
  |P_max - P_t(a_i)| <= epsilon_abs   绝对进度差
  |P_max - P_t(a_i)| / P_max <= epsilon_rel  相对进度差
```

### 组件三：前驱归因

很多危险不是最后一步造成的——真正的因在更早的决策。

```
对历史窗口 W 内每步 t'：
  替换 t' 的动作为最安全等价动作
  测量替换后下游总足迹的降低量 delta_hazard(t')

R_t' = delta_hazard(t') / sum(all deltas)  归一化
w_t = Delta_t * (1 + rho * R_t)  -- 前驱步损失权重上调
```

让模型学习在早期就规避潜在危险，而不只是临近危险时反应。

### 组件四：程序化场景模板

四种可重复的场景注入，系统测试特定风险下的安全偏好：

| 模板 | 测试风险 |
|---|---|
| collateral_clutter | 抓取时附带碰撞 |
| support_critical_neighbor | 平衡临界物体触碰 |
| chain_reaction | 多米诺连锁倒塌 |
| occluded_remembered_hazard | 视野外危险的记忆规避 |

## 4. 训练信号完整路径

```
离线挖掘 (tmp/capra_cache/*.json)
    每步 t 的 q_hat_t, w_t, K 候选动作
         ↓
finetune_capra.py:
    从 RLDS 读 batch（图像+动作+语言）
    从 mining cache 读 capra_records
         ↓
    前向传播 → 预测动作 a_hat
    L_anchor = L1(a_hat, a_demo)  始终计算
         ↓
    若 capra_enabled 且 warmup 结束 且 E_t 非空:
      pi_theta = softmax(-gamma * ||a_i - a_hat||_1)
      L_capra  = w_t * KL(q_hat_t || pi_theta)
         ↓
    L = L_anchor + lambda * L_capra
    反向传播 → 更新参数
```

## 5. 指标设计逻辑

**SPIR**：只关注"能完成任务时是否选了最安全方式"，
独立于任务难度，直接衡量安全偏好。

**EAR**：连续量（而非 topple_rate 的 0/1），
可以检测到"倾向改善"即使还没完全消除倾倒事件。

**SPIR=EAR=0 的正确解读**：
- capra_eval_K=0 时这是正确的空值，不是好结果
- 要比较两个模型的 SPIR/EAR，两者都需要用 capra_eval_K>=2

## 6. 关键工程决策理由

| 决策 | 理由 |
|---|---|
| 离线挖掘而非在线 | 在线 CF rollout 每步需要 K 次仿真重置，太慢 |
| KL 散度而非 reward shaping | KL 方向明确，reward shaping 需要精细设计 |
| 候选集评分近似 | L1 回归头只产生点估计，无法计算精确动作密度 |
| 保留 L_anchor | 确保任务性能不受损，CAPRA 只是额外约束 |
| shuffle_buffer_size=2000 | 更高值导致 TF RLDS 加载器耗尽容器内存 |
| 测试时不加滤波器 | 保持推理延迟不变，安全性由训练偏好保证 |

## 7. 局限性与展望

### 当前局限

1. **snapshot/restore 依赖**：完整 CF 评估需要
   `env.sim.get_state()/set_state()`，不是所有环境都支持。

2. **任务进度代理粗糙**：当前用 done flag（0.5/1.0）估计进度，
   更细粒度的进度函数会提升 E_t 质量。

3. **footprint 信号近似**：I_t 用 cfrc_ext 代理，
   R_t 用角度阈值检测，都是近似，可能误检。

4. **前驱归因计算代价**：完整归因需要多次 rollout，
   当前仅在离线挖掘阶段运行。

### 未来方向

- 更精确的任务进度估计器（替换 done flag）
- 支持 Diffusion 动作头的 CAPRA 损失近似
- 在真实机器人上验证 fidelity=exact 的 snapshot/restore
- 把四种程序化模板扩展为更大的风险场景库
